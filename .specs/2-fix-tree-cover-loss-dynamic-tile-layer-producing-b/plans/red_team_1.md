## Detailed Findings

### 1. Missing Requirement: Analytics Data Independence (C4)

**Severity: HIGH**

The PROMPT.md explicitly states: "Analytics data (statistics, charts) must continue to work independently of the map fix." The master plan does not mention analytics data at all. There is no:
- Analysis of how analytics data flows through the system
- Assertion that the changes do not affect analytics
- Test (automated or manual) verifying analytics still works

**Recommendation:** The plan should include at minimum a paragraph explaining why the changes (session state caching in `render_stream()`, geometry fetch in `render_dataset_map()`) do not affect the analytics pipeline. If analytics data flows through the same `render_stream()` function, the plan must explicitly demonstrate that the modifications do not alter that flow. A manual verification step ("query forest loss and verify statistics/charts still appear") should be added.

### 2. Ambiguity: Zoom Level Selection

**Severity: MEDIUM**

The plan uses `zoom_start = 5` as a hardcoded fallback when geometry is successfully resolved. The Risk Register acknowledges "Zoom level 5 not appropriate for all AOI sizes" but defers to follow-up. However:

- For small AOIs (a city, a small island), zoom 5 may be too zoomed out to see detail.
- For very large AOIs (Russia, Brazil), zoom 5 may be too zoomed in.
- The plan says "The zoom level is calculated from geometry bounds, not hardcoded at 5" in the risk register, but the actual code in Change 1b shows `zoom_start = 5` as the value set after computing center from bounds. There is no dynamic zoom calculation.

A developer implementing this could interpret the plan two ways: (a) always use zoom 5, or (b) compute zoom from bounds. The code shown says zoom 5 always, but the risk register implies dynamic calculation. This is contradictory.

**Recommendation:** Either implement `folium`'s `fit_bounds()` to dynamically set zoom, or explicitly document that zoom 5 is a deliberate simplification and why it is acceptable for the current use cases.

### 3. Ambiguity: Session State Key Scoping

**Severity: LOW-MEDIUM**

The plan uses `st.session_state["last_aoi_data"]` as a flat key. The Risk Register notes "Session state `last_aoi_data` holds stale AOI from a different thread" and suggests scoping by thread_id as a follow-up. However, if the application supports multiple concurrent analysis threads within a single Streamlit session, a stale AOI from a previous query could cause the dataset map to render at the wrong location.

The plan does not clarify whether the application supports multiple concurrent threads/conversations. If it does, this is a real bug, not a deferred concern.

### 4. Gap: No Automated Test for Geometry Fetch Integration

**Severity: LOW-MEDIUM**

Test Suite 1 mocks `ZenoClient.fetch_geometry`. This is appropriate for unit testing, but there is no integration test or smoke test that verifies the actual `fetch_geometry` call works with real AOI data. If the `fetch_geometry` API changes its response shape (e.g., returns `{"geojson": ...}` instead of `{"geometry": ...}`), the unit tests would still pass but the fix would be broken.

**Recommendation:** Add a note about integration testing against a staging API, or add a test that validates the expected response shape from `fetch_geometry`.

### 5. Potential Issue: Duplicate API Call

**Severity: LOW**

The plan introduces a `fetch_geometry()` call in `render_dataset_map()` (Change 1b) that duplicates the call already made in `render_aoi_map()`. Since both functions are called in sequence during `render_stream()`, the same geometry is fetched twice. The plan acknowledges this indirectly ("Extra `fetch_geometry()` call adds latency") but rates it as "Low severity, Medium likelihood."

A more elegant approach would be to cache the geometry result in session state during the `render_aoi_map()` call and reuse it in `render_dataset_map()`. The plan's Change 1a caches `aoi_data` (which does NOT contain geometry), so the geometry is still fetched again. This is a missed optimization.

### 6. Minor: Sidebar Fix Priority

**Severity: LOW**

Change 2b fixes sidebar hardcoded data (`dataset_id=0` to `4`, threshold `25` to `30`). The plan marks this as P2 but the sidebar is a user-facing feature. If users select TCL from the sidebar, the wrong dataset_id could cause the wrong dataset to load entirely (dataset 0 is DIST-ALERT, not TCL). This seems more like P1 severity.

---

## Summary Assessment

**Overall Coverage: 9 of 10 requirements addressed.** One constraint (C4 -- analytics data independence) is completely missing from the plan.

**Testability: GOOD but incomplete.** Automated tests cover URL construction and geometry resolution. Layer order and end-to-end rendering rely on manual verification only.

**Implementability: HIGH.** The plan provides specific file paths, line numbers, before/after code snippets, and implementation order. A developer could follow this with minimal interpretation.

**Key Risk: The zoom_start=5 hardcode is both the fix and a potential new problem.** The entire fix relies on the assumption that zoom 5 produces visible TCL tiles. The pre-implementation curl verification step is good, but if zoom 5 is insufficient for certain AOIs, the user will still see a blank-looking map. Dynamic zoom calculation (via `fit_bounds`) would be more robust.

**Verdict:** The plan is solid and well-structured. The two critical items to address before implementation are: (1) add analytics data independence verification, and (2) clarify the zoom level calculation ambiguity in the code vs. risk register.
